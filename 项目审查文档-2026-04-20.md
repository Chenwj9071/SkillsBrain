# 项目审查文档 - 2026-04-20

## 一、项目目标理解

该工程是一个**本地技能路由引擎**，核心目标包括：

1. 扫描本地 `skills/` 目录下的 `SKILL.md`
2. 解析技能元数据
3. 使用向量模型进行语义索引与检索
4. 通过 API / CLI 为 AI Agent 返回最匹配的技能
5. 支持文件变更监听，自动增量更新索引

整体定位可以理解为一个“**本地版 skill registry + semantic router**”。

从架构上看，分层基本清晰：
- `parser`：技能元数据提取
- `indexer`：向量索引构建
- `engine`：检索流程
- `api`：服务暴露
- `watcher`：增量同步
- `cli`：命令行入口

结论：

> 工程方向正确，但当前实现仍更接近“可演示 / 原型阶段”，距离“稳定可用、可维护、可审计”的工程状态还有明显差距。

主要问题集中在：
- 数据一致性
- 删除/更新逻辑
- 权限边界
- 错误处理
- API 设计安全性
- 打包结构一致性
- 配置与行为一致性

---

## 二、总体结论

### 优点

1. **模块边界基本清楚**  
   `parser/indexer/engine/api/watcher/cli` 分层自然，便于后续演进。

2. **支持本地化部署**  
   不依赖外部检索服务，适合离线或私有环境。

3. **增量监听思路正确**  
   使用 watchdog 自动刷新索引，符合 skill registry 的使用习惯。

4. **检索链路简单直接**  
   “召回 + 过滤 + 阈值排序”易于落地和调试。

### 总体判断

> 当前版本适合原型验证、功能演示和小规模本地试验；不建议直接作为稳定生产工具投入使用。

主要原因：
1. 删除/重建索引不正确，容易产生脏数据
2. 兼容性过滤失败后回退未过滤结果，破坏业务语义
3. 管理接口无鉴权且暴露绝对路径，存在安全隐患

## 四、P0 / P1 级问题明细

### P0-1：删除技能的逻辑错误，导致索引残留脏数据

**位置**：
- `src/skillsbrain/core/indexer.py`
- `update_skill`
- `delete_skill`

**问题代码**：
```python
name = file_path.stem
self._collection.delete(ids=[name])
```

对于 `skills/pdf/SKILL.md`，`file_path.stem == "SKILL"`，但插入时使用的 id 却是：
```python
ids = [s.name for s in skills]
```

也就是说：删除时用的是 `SKILL`，插入时用的是 metadata 中的 `name`，两者通常不一致。

**后果**：
- 删除文件时，索引不会真正删除对应 skill
- 修改失败时的兜底删除也删不掉
- 已删除技能仍可能被检索到
- 索引会长期积累脏数据

**建议**：
必须建立**稳定且可逆的主键策略**。

推荐方案：
- 使用技能文件的**相对路径**或目录名作为稳定 id
- 不要只使用 `metadata.name` 作为唯一索引主键

例如：
- `skills/pdf/SKILL.md -> id=pdf`
- 或 `id=pdf/SKILL.md`

---

### P0-2：`full_sync()` 不是“全量同步”，会保留历史脏数据

**位置**：`src/skillsbrain/core/indexer.py`

当前 `full_sync()` 只是扫描后 upsert：
```python
if skills:
    self._upsert(skills)
```

**问题**：
没有删除索引中已经不存在的条目，因此并不是严格意义上的“全量重建”。

**后果**：
- 文件删除后索引仍保留
- 文件重命名后旧数据残留
- `/reindex` 不能真正重建索引
- README 描述与实现不一致

**建议**：
实现真正的全量同步：
1. 扫描当前文件系统，得到当前 skill id 集合
2. 读取 collection 现有 id 集合
3. 删除不存在于文件系统中的 stale ids
4. 再执行 upsert

或者提供：
- `reset/rebuild`：先清空 collection，再重建

---

### P0-3：过滤失败后回退未过滤结果，破坏兼容性和 enabled 语义

**位置**：`src/skillsbrain/core/engine.py`

问题代码：
```python
filtered = self._filter_by_metadata(candidates, agent_type=agent_type)
if not filtered:
    logger.info("Metadata filter emptied results, returning top unfiltered candidates.")
    filtered = candidates[:settings.top_k_recall]
```

**问题**：
当指定 `agent_type` 但没有兼容技能时，系统会回退返回“未过滤结果”。

**后果**：
- 返回不兼容目标 Agent 的技能
- disabled 技能在某些场景下可能被放回结果集中
- 兼容性过滤语义失真，不可信
- 上层编排器可能收到错误技能，导致执行失败或误判系统能力

**建议**：
- 过滤后为空时，**直接返回空列表**
- 如确有调试需求，可增加显式开关：
  - `strict_compatibility = true` 默认开启
  - 仅调试模式允许 fallback

---

### P1-4：`enable/disable` 只改索引，不改源文件，状态不可持久

**位置**：`src/skillsbrain/api/main.py`

当前 `/api/skill/enable/{name}` 只更新 Chroma metadata：
```python
meta["enabled"] = str(enabled).lower()
```

**问题**：
- 不会回写 `SKILL.md`
- 文件一旦变更，watcher 重新索引后会覆盖该状态
- 服务重启或重建索引后状态会丢失

**后果**：
API 看上去像“持久修改”，实际上只是“临时改索引状态”。

**建议**：
三选一：
1. 在 API 与文档中明确声明这是**临时状态**
2. 正式实现 frontmatter 回写
3. 单独维护 override store（如 sqlite/json）

当前实现属于危险的半持久状态设计。

---

### P1-5：API 暴露绝对路径 `file_path`，存在信息泄漏风险

**位置**：
- `parser.py`
- `api/main.py`

当前 `file_path` 被设置为绝对路径并直接通过 API 返回。

**风险**：
- 泄露宿主机目录结构
- 泄露用户目录、磁盘信息
- 为攻击者提供部署细节

**建议**：
- API 默认只返回 `relative_path`
- 绝对路径仅在 debug/admin 模式可见

---

### P1-6：CORS 配置过宽，且与 credentials 并用不安全

**位置**：`src/skillsbrain/api/main.py`

当前配置：
```python
allow_origins=["*"],
allow_credentials=True,
allow_methods=["*"],
allow_headers=["*"],
```

**问题**：
这属于典型宽松配置，不适合作为默认生产配置。

**建议**：
- 默认仅允许 `localhost/127.0.0.1`
- 管理接口不要默认跨域
- 将 CORS 改为配置项

---

### P1-7：管理接口没有鉴权

**涉及接口**：
- `/api/skill/enable/{name}`
- `/api/skill/reindex`
- `/api/skill/list`
- `/api/skill/detail/{name}`
- `/api/skill/stats`

**问题**：
当前仅依赖“默认绑定 127.0.0.1”作为边界，这不构成真正的安全策略。

**风险**：
本机任意恶意进程都可以：
- 枚举技能
- 获取路径信息
- 反复触发 reindex 造成资源消耗
- 修改 skill enabled 状态

**建议**：
- 区分 public API 与 admin API
- admin API 增加 token 鉴权
- `/reindex`、`/enable` 必须鉴权

---

### P1-8：Parser 吞掉全部异常，故障不可观测

**位置**：`src/skillsbrain/core/parser.py`

当前：
```python
except Exception:
    return None
```

**问题**：
任何 frontmatter 错误、编码错误、字段类型错误都会被静默吞掉。

**后果**：
- skill 丢失但无日志
- watcher 更新失败没人知道
- 运维与开发无法定位原因

**建议**：
至少记录异常：
```python
except Exception as e:
    logger.exception("Failed to parse skill file: %s", file_path)
    return None
```

---

### P1-9：`enabled=bool(...)` 存在布尔解析错误

**位置**：`src/skillsbrain/core/parser.py`

当前：
```python
enabled=bool(km.get("enabled", True))
```

**问题**：
若 frontmatter 写成：
```yaml
enabled: "false"
```
则：
```python
bool("false") == True
```

**后果**：
禁用状态会被误判为启用。

**建议**：
做严格布尔解析，正确处理 `bool / str / int` 类型。

---

### P1-10：`compatibility` 和 `tags` 未做标准化，可能污染索引

**问题**：
若 frontmatter 中写成字符串而不是数组，例如：
```yaml
compatibility: claude_code
tags: pdf
```
则后续：
```python
",".join(s.compatibility)
```
可能变成字符级 join。

**后果**：
- metadata 污染
- 过滤失败
- API 返回异常字段内容

**建议**：
在 parser 层统一标准化为 `list[str]`。

---

### P1-11：`created_at` 未做格式校验

**问题**：
当前只是字符串透传，没有日期格式校验，也没有时区语义。

**建议**：
- 统一 ISO8601 格式
- parser 层校验合法性
- 文档明确格式要求

---

### P1-12：跨层直接访问 `_collection`，破坏封装

**位置**：
- `engine.py`
- `api/main.py`

**问题**：
多个模块直接访问 `indexer._collection`，依赖了存储层内部实现。

**后果**：
- 难以替换存储层
- 查询逻辑分散
- 易出现多处不一致行为

**建议**：
为 `SkillIndexer` 提供正式接口，如：
- `query()`
- `list_skills()`
- `get_skill()`
- `set_enabled()`
- `delete_by_file()`

---

### P1-13：`top_k` 参数设计混乱

**位置**：`src/skillsbrain/core/engine.py`

`_semantic_recall(self, query, top_k)` 中传入了 `top_k`，但内部实际使用的是：
```python
recall_k = settings.top_k_recall
```

**问题**：
传参与实现不一致，构成误导。

**建议**：
- 要么删除该参数
- 要么真正使用它

---

### P1-14：`device` 配置未生效

**位置**：`src/skillsbrain/config.py` 与 `src/skillsbrain/core/indexer.py`

配置中声明了：
```python
device: str = "cpu"
```
但模型加载时未使用。

**建议**：
```python
SentenceTransformer(settings.embedding_model, device=settings.device)
```

## 五、P2 / P3 级问题

### P2-15：项目存在两套并行源码，结构混乱

目录中同时存在：
- `src/skillsbrain/...`
- 根目录 `api/ core/ utils/ ...`

**风险**：
- 开发者不知道该修改哪套代码
- 导入路径混用
- 审查与维护成本增加

**建议**：
统一保留 `src/skillsbrain/**`，删除或归档旧版平铺代码。

---

### P2-16：CLI 错误处理不完整

当前 CLI 只处理 `ConnectionError`，没有覆盖：
- timeout
- 4xx/5xx 返回信息
- 非 JSON 响应

**建议**：
统一封装请求层，补充错误处理与用户提示。

---

### P2-17：日志初始化可能重复注册 handler

`create_app()` 中执行日志初始化，如果 app 被多次创建，可能造成 handler 重复问题。

**建议**：
封装日志初始化逻辑，并检测 logger 是否已配置。

---

### P2-18：watcher 每次事件启动一个线程，线程模型粗糙

当前每个文件事件都：
```python
threading.Thread(target=_flush, daemon=True).start()
```

**风险**：
- 大量变更时创建过多线程
- 并发顺序不可控
- 资源浪费

**建议**：
使用：
- 单 worker + queue
- 或集中 debounce 调度器

---

### P2-19：缺少 shutdown 生命周期处理

watcher 启动后没有在 shutdown 时优雅关闭。

**建议**：
- 将 observer 保存到 `app.state`
- 在 shutdown 生命周期中 `stop + join`

---

### P2-20：调用日志记录 query，缺少脱敏与关闭策略

`call_logger` 会记录：
- query
- session_id
- hits

**风险**：
query 中可能含敏感信息。

**建议**：
- 支持关闭调用日志
- 支持脱敏 / 截断
- 明确日志内容说明

---

### P3-21：检索流程描述与实现不一致

README 声称是“过滤 → 召回 → 精排”，但实际实现是“召回 → 过滤 → 阈值排序”。

**建议**：
- 修正文档
- 或按文档真正实现 metadata 预过滤

---

### P3-22：所谓“精排”并不是真正 rerank

当前只是阈值筛选 + 排序，不属于严格意义上的 reranker。

**建议**：
README 改写为“阈值过滤与排序”。

---

### P3-23：索引文本过于简单，影响召回质量

当前 embedding 只使用：
- `name`
- `description`
- `tags`

**建议**：
可扩展支持：
- `examples`
- `aliases`
- `category`
- `tooling`
- `use_cases`

---

### P3-24：缺少 readiness 检查与分页

**建议**：
- 增加 `/health/ready`
- `list` 接口增加分页能力

---

### P3-25：README 与 requirements / pyproject 存在不一致

**建议**：
- 同步 `requirements.txt` 与 `pyproject.toml`
- 校正文档描述与实际行为不一致问题

## 六、按优先级输出的修改建议列表

### 第一优先级：必须立即修复

1. **重构技能索引主键设计**  
   使用稳定、可逆的 skill id（推荐相对路径或目录名），彻底修复删除、更新、重建一致性问题。

2. **修复 `full_sync()` 逻辑**  
   实现真正的“全量同步/重建”，在重建时清理 stale 数据。

3. **移除过滤失败后的未过滤 fallback**  
   `agent_type` / `enabled` 过滤后为空时应直接返回空结果。

4. **修复删除逻辑**  
   watcher 删除事件必须能删除真实 skill id 对应的索引，而不是错误删除 `SKILL`。

5. **为 parser 增加严格校验与异常日志**  
   修复 `enabled`、`compatibility`、`tags` 等字段类型问题，并保证错误可观测。

6. **管理接口增加鉴权**  
   至少为 `/reindex`、`/enable` 增加 token 保护。

7. **默认不暴露绝对路径**  
   API 改为返回相对路径，绝对路径只允许在 debug/admin 模式查看。

---

### 第二优先级：强烈建议尽快修复

8. **明确 `enable/disable` 的持久化策略**  
   选择“临时状态”“回写 frontmatter”或“override store”三种方案之一，不要停留在半持久状态。

9. **收敛存储层封装**  
   不再跨层直接访问 `_collection`，将查询/列举/更新能力统一封装到 `SkillIndexer`。

10. **统一代码结构，移除旧版平铺源码**  
    只保留 `src/skillsbrain/**` 一套实现。

11. **收紧 CORS 默认配置**  
    避免使用 `*` 与 `credentials=true` 组合。

12. **为 watcher 增加 shutdown 管理**  
    在应用关闭时优雅停止 observer。

13. **优化 watcher 并发模型**  
    由“每事件一个线程”改为更稳定的队列/调度方案。

14. **让 `device` 配置真正生效**  
    加载模型时使用 `settings.device`。

15. **修复 `top_k` 参数设计不一致问题**  
    保证接口、配置与实现行为一致。

---

### 第三优先级：建议作为增强项持续完善

16. **完善 CLI 错误处理与输出**  
    补充 timeout、HTTP 错误、JSON 错误场景。

17. **为调用日志增加脱敏、截断与关闭能力**

18. **增加分页与 readiness 健康检查接口**

19. **优化索引文本构造，提升召回质量**

20. **修正文档与实现不一致问题**  
    包括 README 中关于“三层检索”“重建索引”等描述。

21. **同步依赖清单**  
    保证 `requirements.txt` 与 `pyproject.toml` 一致。

22. **清理未使用代码和无效 import**  
    如 `yaml` 未使用等。

---

## 七、建议的整改方向

如果作为后续重构路线，建议按以下方式推进：

1. 定义稳定 Skill ID
2. 在 parser 层做强校验和类型标准化
3. 在 indexer 层统一封装 Chroma 访问
4. 在 engine 层保证检索语义严格可信
5. 在 API 层区分 public/admin 能力并增加鉴权
6. 明确索引状态与源文件状态的职责边界

---

## 八、最终审查结论

> 该项目具备清晰方向和基本原型价值，但当前实现仍存在多处关键缺陷，不适合直接作为稳定生产能力交付。

最严重问题集中在：
- 索引删除与重建不正确
- 检索过滤语义被错误 fallback 破坏
- 管理接口安全边界不足

建议优先完成 P0 / P1 整改后，再评估是否进入团队级稳定使用阶段。
